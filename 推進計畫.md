## 4. P6.5｜花曆上雲（獨立設計）

> 命名說明：本文件統一用 P0–P9 序號。花曆＝**P6.5**、預約＝**P8**，兩者都不使用「3.6」，以免與 GPT-5 原 Phase 3.x 家族混淆。

**決定：花曆要可更改，但用表格分頁、不塞進 708 扁平欄位，且排在核心後台穩定後。**

現況：`seasons.js` 的 `FLOWERS` 陣列寫死、目前僅 24 筆（完整 90+ 筆未補齊），程式已註明「第二階段改由試算表讀取」。畫面只是自動高亮當月。

做法：
- 在後台 Sheet 新增一個「花曆」分頁，**一列一種植物**，欄位：花名 / 花期(month) / 類型(花卉·果樹·香草·蔬菜·觀葉) / 描述 / 圖片網址。
- 前端讀成陣列取代 `FLOWERS`；甘特圖與篩選邏輯不變。
- 不用「一 key 一列」的 flat 模型（90×5≈450 格會拖垮主後台），tabular 表格更適合這種重複列資料。

排序理由：花況一年才校一次、屬低頻，不該擋主線；等 P3 接通、P5 實測後再做。

## 5. 預約功能方向（P8 之後）

- **第一版價值 = 把 FB／電話／LINE／網站來源匯進同一本帳**，不是客人自助付款。
- 資料放**獨立私有 Sheet**（限定共用），**獨立 Apps Script 部署**，驗證比內容後台更嚴（存姓名/電話/匯款末五碼等個資，POST-only、不開放公開讀取）。
- 30 筆/月的量：做**內部補登工具**值得；**不做**客人自助訂位/即時桌位/庫存。
- 狀態流程：待聯繫 → 已報名待訂金 → 已收訂 → 完成/取消；提供催訂金清單。
- 補登欄位：預約類型、活動日期、人數、姓名、聯絡方式、來源、訂金、收款方式、匯款末五碼、收款日期、備註、狀態。
- 預約類型：DIY/校外教學、生態導覽、婚禮/包場、團體聚餐、家庭聚餐。

**金流：延後。** 三條件全成立才做：(a) 補登流程順跑完一整個旺季；(b) 量大到人工核對末五碼真的痛；(c) 老闆/會計談定錢落哪、誰處理發票/退款/取消。屆時**優先走 SO-BUY「活動訂金商品」**（複用現有商家、免新申請金流），而非直接串閘道。

- 內容：接通 Phase 3、後台預設常用內容、SEO 欄位人工鎖定＋checklist、可見頁價格單一來源。

**未來完整（Full）**

- 預約：客人網頁表單餵同一流程、SO-BUY 訂金整合、報表/催訂金自動化。
- 金流：SO-BUY 先行，量夠再談閘道。
- 主要風險：自動化寫 GitHub 的安全面、金流/個資/發票合規、照現況量做過頭。
