## 核心原則：自建 vs 直接買現成

「完整農場平台」全部自建是維運無底洞。單一農場、旺季約 30 筆/月團體預約的量，多數模組**買現成 SaaS 或用既有工具，遠比自建划算**。判準：

- **獨特、且是你們的核心流程** → 值得自建（例：把所有來源匯進同一本預約帳）。
- **通用、已有成熟方案（訂房、訂位、金流、代辦）** → 買現成，別重造輪子。
- **每一個自建模組 = 一份長期維運責任**（bug、資安、個資、對帳），先問「有沒有人長期維護」。

## 模組清單與建議

### A. 統計 / 資料彙整　🟢 最先做、最划算（自建，容易）
- P8 預約進 Sheet 後，報表幾乎免費。**兩種轉換率要分清楚，資料來源不同：**
  - **申請→收訂率／成團率**：可直接用預約 Sheet 算（有申請、收訂、成團欄位即可）。
  - **FB／網站訪客→預約率**：**Sheet 算不出來**，需要 **UTM 參數 + GA4／Meta Pixel** 才能把「看到的人」對上「來預約的人」。所以想看廣告成效，追蹤要先裝（見 `推進計畫.md` P7.5）。
- 其他彙整：依來源（FB/電話/LINE/官網）、類型、月份、訂金收款。
- 做法：Sheet 樞紐分析 + 圖表，或一頁儀表板。價值高、風險低，第二階段第一個做。

### B. 客人自助預約表單（DIY / 導覽）　🟡 值得自建（中等）
- 「日期 + 場次 + 人數 + 類型」的申請，是 P8 內部補登的延伸：官網表單 → 同一本私有預約 Sheet → 人員確認。
- **先不做即時名額鎖定**（30 筆/月不需要）；先做「申請 → 人工確認」。
- **個資與防濫用（開表單前必須先設計好）：**
  - **告知**：表單標明蒐集目的、範圍、保留期限與聯絡方式（符合個資法告知義務）。
  - **最小化蒐集**：只收必要欄位（姓名、聯絡方式、日期、人數、類型），不多要。
  - **存取權限**：預約 Sheet 限定人員存取、不可公開分享連結。
  - **保留期限**：訂定活動結束後多久封存／刪除個資。
  - **防垃圾送單**：honeypot 隱藏欄位、reCAPTCHA、限流或必填驗證，避免機器人灌單。

### C. 匯款確認與線上付款　🟡 中等（拆兩步）

**2C-1 匯款確認 Email（先做）**
- 人員在後台標「已收訂」→ Apps Script 自動寄確認信（`MailApp`）。
- **前提**：預約表單/Sheet 要有**明確的 Email 欄位**，且**先確認客人是否常提供 Email**——農場多從 FB/電話接單，可能很多客人沒給 Email。若普遍沒有，自動 Email 效益低，通知改走人工 LINE 回覆或簡訊，再評估。
- 注意：LINE Notify 已於 2025 年停用；要 LINE 通知需改用 LINE Messaging API（較費工），初期先 Email 或人工。

**2C-2 第三方線上付款（後做）**
- 走第三方 hosted checkout（**SO-BUY 訂金商品**優先，或綠界/藍新）。GitHub Pages 靜態站不能自己收款。
- **不可預設有 webhook 自動回填**：**只有平台官方支援自動回填、且文件完整時才串**；否則採**訂單編號人工核對**（客人付款後回填訂單編號，人員比對）。
- 串金流的前提仍是第 5 節（`推進計畫.md`）的金流三條件成立。

### D. 餐廳訂位 / 訂桌　🔵 建議買現成，別自建
- **注意：旺季 30+ 筆是「團體活動預約」，不等於餐廳訂桌量。** 餐廳要不要導系統，應**另依實際訂桌量、漏接電話、撞桌／重複訂位的情況**判斷。
- 即時桌位/時段管理是複雜獨立問題，量不大時自建不划算。
- 選項：維持電話/LINE；或現成訂位 SaaS（inline、EZTABLE 等）；或 Google「透過 Google 訂位」。仍不建議自建。

### E. 住宿預約（星空露營等，籌備中）　🔵 建議買現成
- 可訂日曆、避免超賣是成熟問題，交給訂房平台或民宿訂房 SaaS。
- 房量極少時，先做「申請表 + 人工確認」，不要自建訂房引擎。

### F. 人力分配 / 代辦事項　🔵 建議用既有工具（輕度自建）
- 小團隊不需完整排班系統。用現成（Google Tasks / Trello / Notion）即可。
- 若要跟預約綁在一起：在預約 Sheet 加「待辦」欄位——每筆預約的動作（催訂金、確認人數、派導覽員/場地）+ 負責人 + 期限。輕度自建、綁在 B 之上，勝過獨立系統。

## 建議推進順序（第二階段內部）

1. **2A 統計/報表**（易、高值；P8 穩定後即可）。
2. **2B 客人自助預約表單**（DIY/導覽）餵同一 Sheet + 人工確認（含個資與防濫用設計）。
3. **2C-1 匯款確認 Email**（先確認客人有無 Email）。
4. **2C-2 第三方線上付款**（SO-BUY 訂金優先；有官方自動回填才串，否則訂單編號人工核對）。
5. **2D 代辦/人力**（用既有工具或綁在預約 Sheet 上）。
6. **2E 訂位/住宿**——依實際量導入現成 SaaS，不自建。

## 架構天花板提醒

- 現行 **Apps Script + Google Sheet + GitHub Pages** 撐內容後台與輕量預約補登沒問題。
- 但「完整平台」若要**即時可訂庫存（住宿/桌位）+ 金流 + 多通路通知 + 多人協作**，會超出這套的舒適圈。屆時兩條路：
  - **(推薦) 拼接現成 SaaS**：訂房/訂位/金流/代辦各用成熟工具，官網只當入口與導流；維運外包、風險低。
  - **(重) 自建真正的應用**（前後端 + 資料庫 + 帳號系統）：只有在有獨特流程、且有預算與長期維護人力時才做。
- 單一農場的結論通常是：**核心資料彙整自建、其餘買現成拼接**。
