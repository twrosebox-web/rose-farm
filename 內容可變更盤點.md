# 大花農場體驗頁｜內容「可變更 vs 寫死」盤點

> 用途：讓你一眼看出哪些內容農場人員能從後台改、哪些必須動程式碼、哪些同時存在兩份要小心同步。
> 依據：`data.js`、`data-modals.js`、`cloud-data.js`、`seasons.js`、`index.html` 實際內容（2026-07）。
> 標示：✅ 可雲端變更　🔒 寫死（改要動程式）　⚠ 雙份需手動同步

---

## 三、⚠ 雙份需手動同步（同一資訊存在兩處，改一處另一處不會動）

> 這是目前最大的內容風險。後台改了雲端版、畫面會更新，但寫死在 index.html 的靜態版（SEO 用）仍是舊值。

| 資訊 | 雲端版（會變） | 靜態版（不會變，要人工改） |
|---|---|---|
| **票價** | `siteConfig.ticket.*` → 參觀資訊、餐廳提示 | JSON-LD offers + noscript FAQ 內的價格 |
| **FAQ 問答** | `qa.*` → FAQ 區塊 | JSON-LD FAQPage + noscript 9 題 `<article>` |
| **地址／電話／營業時間** | `siteConfig.*` | JSON-LD address / telephone / openingHours |
| **主標題 H1** | `pageContent.hero.title` | `<title>` 標籤（獨立寫死） |

**建議：** 這 4 類約 15–20 個 key 列為「SEO 敏感」，一般人員視圖設唯讀或掛「改此欄需同步靜態頁」警告；改動時走一張固定 checklist。因為這些實際變動頻率很低（票價、地址、FAQ 很少改），人工同步成本可接受。

## 四、需實測確認（不確定，建議接通後台後驗證）

雲端載入完成後，`cloud-data.js` 只會重跑這幾支模組：`elevator / bento / eco-services / dining / products-diy / info-faq`。
**沒有重跑** `hero / features / seasons / gallery`，這幾支只在初次載入時渲染。

- 圖片：這幾區的圖片有另外用 `renderStaticImages()` 直接更新 DOM，**應該會變**。
- 文字：`hero / features / seasons四季卡 / gallery` 的**文字**（如四季卡名稱/描述、特色標題）在雲端覆蓋 `DATA` 後，模組沒重跑，畫面文字可能**不會即時更新**。

→ 接通後台後，實際改一次「四季卡描述」或「三大特色標題」測試是否生效。若不生效，需把這幾支也加入重繪清單，或改成 `renderStaticContent` 一併處理。

## 五、花曆的決定點

你提到花曆「設定好季節會自動變」——實際上畫面是自動高亮當月，但**底層 24 筆植物資料是寫死且不完整的**。三個選項：

1. **維持現狀**：花曆當「設定一次」的內容，未來要改文字/圖片時再由工程處理。
2. **補齊後仍寫死**：先把 90+ 筆補完整，但仍放 seasons.js，農場不需自行維護。
3. **納入雲端**（符合你「全可調」初衷）：把 FLOWERS 移到後台欄位，欄位數會再增加約 90×5 欄，但花況可自行維護。

建議：**先選 1 或 2**（花況一年才校一次，不值得為它擴充上百欄）；等後台整體上線穩定、真的有維護需求再考慮 3。
